## 二、i18n 国际化技术方案

### 2.1 前端：next-intl

**选型理由**：与 Next.js App Router 原生集成，支持 Server Component 和 Client Component，配置简洁。

**文件结构**：

```
frontend/
  ├── packages/ui/
  │     └── locales/
  │           ├── zh-CN.json    # 中文翻译
  │           └── en.json       # 英文翻译
  ├── apps/web/
  │     ├── i18n/
  │     │     ├── request.ts    # getRequestConfig 配置
  │     │     └── routing.ts    # 路由国际化配置
  │     ├── middleware.ts       # 语言检测中间件
  │     └── app/
  │           └── [locale]/     # 动态语言路由段
  │                 ├── layout.tsx
  │                 ├── page.tsx
  │                 ├── (auth)/
  │                 └── (account)/
```

**核心配置**：

1. **`middleware.ts`**：读取 cookie `NEXT_LOCALE`，匹配请求路径，设置 locale
2. **`i18n/request.ts`**：`getRequestConfig` 根据 locale 加载对应的翻译文件
3. **翻译文件格式**（以 `zh-CN.json` 为例）：

```json
{
  "auth": {
    "login": { "title": "欢迎回来", "username": "用户名", "password": "密码", "submit": "登录" },
    "register": { "title": "创建新账号", "username": "用户名", "email": "邮箱", "password": "密码", "passwordRetype": "确认密码", "code": "验证码", "sendCode": "发送验证码", "submit": "注册" }
  },
  "common": {
    "language": "语言",
    "switchLang": "切换语言"
  }
}
```

4. **组件中使用**：
   - Server Component：`const t = await getTranslations('auth.login')` → `t('title')`
   - Client Component：`const t = useTranslations('auth.login')` → `t('title')`

5. **语言切换组件**：右上角下拉菜单，切换时设置 cookie `NEXT_LOCALE` 并刷新页面

### 2.2 后端：Django i18n

**配置变更**（`backend/api/settings.py`）：

```python
LANGUAGE_CODE = 'en'  # 默认语言

LANGUAGES = [
    ('en', 'English'),
    ('zh-cn', '简体中文'),
]

LOCALE_PATHS = [
    BASE_DIR / 'locale',  # 翻译文件目录
]

# 中间件
MIDDLEWARE = [
    ...
    'django.middleware.locale.LocaleMiddleware',  # 新增，放在 SessionMiddleware 之后
    ...
]
```

**翻译文件生成流程**：

```bash
# 提取可翻译字符串
docker compose exec api uv run -- python manage.py makemessages -l zh_CN
# 翻译完成后编译
docker compose exec api uv run -- python manage.py compilemessages
```

**生成文件**：`backend/locale/zh_CN/LC_MESSAGES/django.po`

**代码中的使用**：

```python
from django.utils.translation import gettext_lazy as _

# Serializer 字段标签
username = serializers.CharField(label=_("用户名"))

# 错误消息
raise serializers.ValidationError({"username": _("该用户名已被注册")})
```

**API 错误消息的语言切换**：

- 前端请求时通过 `Accept-Language` 请求头传递当前语言
- Django `LocaleMiddleware` 自动根据请求头激活对应语言
- DRF 的 `ValidationError` 消息会自动使用激活的语言

### 2.3 Django Admin 国际化

`django-unfold` 自带多语言支持，配置 `LANGUAGES` 后 Admin 界面自动支持中英文切换。Admin 右上角会显示语言切换下拉菜单。

## 三、邮箱验证码注册技术方案

### 3.1 新增数据模型

**文件**：`backend/api/models.py`

```python
class EmailVerification(models.Model):
    """邮箱验证码记录"""
    id = models.UUIDField(primary_key=True, default=uuid.uuid4, editable=False)
    email = models.EmailField(verbose_name=_("邮箱"))
    code = models.CharField(max_length=6, verbose_name=_("验证码"))
    created_at = models.DateTimeField(auto_now_add=True, verbose_name=_("创建时间"))
    is_used = models.BooleanField(default=False, verbose_name=_("是否已使用"))

    class Meta:
        db_table = "email_verification"
        ordering = ["-created_at"]
        indexes = [
            models.Index(fields=["email", "-created_at"]),
        ]
```

**字段说明**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | UUID | 主键，避免自增 ID 被枚举 |
| `email` | EmailField | 目标邮箱地址 |
| `code` | CharField(6) | 6 位数字验证码 |
| `created_at` | DateTimeField | 创建时间，用于判断有效期 |
| `is_used` | BooleanField | 是否已使用，防止验证码重复使用 |

### 3.2 新增 API 接口

#### 3.2.1 发送验证码

- **URL**：`POST /api/email/send-code/`
- **认证**：无需认证（AllowAny）
- **请求参数**：

```json
{
  "email": "user@example.com"
}
```

- **成功响应**（200）：

```json
{
  "message": "验证码已发送"
}
```

- **错误响应**（400）：

```json
{
  "email": ["请输入有效的邮箱地址"]
}
```

```json
{
  "email": ["发送过于频繁，请 60 秒后重试"]
}
```

```json
{
  "email": ["今日发送次数已达上限"]
}
```

- **业务逻辑**：
  1. 校验邮箱格式
  2. 查询 `EmailVerification` 表，检查同一邮箱最近一条记录的 `created_at` 是否在 60 秒内
  3. 统计同一邮箱过去 24 小时内的记录数，若 ≥ 10 则拒绝
  4. 生成 6 位随机数字验证码：`str(random.randint(100000, 999999))`
  5. 写入 `EmailVerification` 表
  6. 调用 `django.core.mail.send_mail()` 发送邮件
  7. 返回成功响应

#### 3.2.2 注册接口改造

- **URL**：`POST /api/users/`（复用现有路由）
- **认证**：无需认证（AllowAny）
- **请求参数**（改造后）：

```json
{
  "username": "testuser",
  "email": "user@example.com",
  "password": "securepass123",
  "password_retype": "securepass123",
  "code": "123456"
}
```

- **改造要点**：
  1. `UserCreateSerializer` 新增 `email` 和 `code` 字段
  2. `email` 字段：EmailField，唯一性校验
  3. `code` 字段：CharField(6)，验证码校验
  4. `create()` 方法中验证验证码：
     - 查询 `EmailVerification` 表，条件：`email=邮箱, code=验证码, is_used=False, created_at__gte=5分钟前`
     - 找不到则抛出"验证码错误或已过期"
     - 找到后标记 `is_used=True`
  5. 创建用户时 `is_active=True`（原来是 `False`）
  6. User 模型需要新增 `email` 字段（Django AbstractUser 自带 `email` 字段，但当前非必填且无唯一约束，需改为必填 + 唯一）

**User 模型改造**：

```python
class User(AbstractUser):
    # 已有字段
    created_at = models.DateTimeField(auto_now_add=True)
    modified_at = models.DateTimeField(auto_now=True)

    # email 改为必填且唯一
    email = models.EmailField(
        verbose_name=_("邮箱"),
        unique=True,
        error_messages={
            "unique": _("该邮箱已被注册"),
        },
    )
```

## 四、RBAC 角色权限技术方案

### 4.1 角色判断依据

基于 Django 内置字段，不引入第三方 RBAC 库：

| 角色 | 判断字段 | 说明 |
|------|----------|------|
| 管理员 | `is_staff=True` | Django 内置字段，Admin 后台访问权限 |
| 普通用户 | `is_staff=False` | 默认值 |

### 4.2 后端权限控制

#### 4.2.1 自定义权限类

**文件**：`backend/api/permissions.py`（新建）

```python
from rest_framework.permissions import BasePermission


class IsOwnerOrAdmin(BasePermission):
    """
    普通用户只能操作自己的数据，
    管理员可以操作所有数据。
    """

    def has_permission(self, request, view):
        return request.user and request.user.is_authenticated

    def has_object_permission(self, request, view, obj):
        if request.user.is_staff:
            return True
        # obj 需要有 user 字段指向创建者
        return getattr(obj, "user", None) == request.user
```

#### 4.2.2 ViewSet 中的应用方式

以未来任务 ViewSet 为例：

```python
class TaskViewSet(viewsets.ModelViewSet):
    permission_classes = [IsOwnerOrAdmin]

    def get_queryset(self):
        # 管理员看全部，普通用户只看自己的
        if self.request.user.is_staff:
            return Task.objects.all()
        return Task.objects.filter(user=self.request.user)
```

#### 4.2.3 现有 UserViewSet 调整

现有 `UserViewSet` 的权限逻辑基本正确，保持 `create` 动作用 `AllowAny`，其余用 `IsAuthenticated`。`me` 动作只返回当前用户信息，无需额外改动。

### 4.3 前端权限控制

#### 4.3.1 Session 扩展

**文件**：`frontend/apps/web/lib/auth.ts`

在 next-auth 的 `jwt` 和 `session` callback 中增加 `is_staff` 字段：

```typescript
// jwt callback
async jwt({ token, user }) {
  if (user) {
    token.isStaff = user.is_staff  // 从 Django API 获取
  }
  return token
}

// session callback
async session({ session, token }) {
  if (session.user) {
    session.user.isStaff = token.isStaff
  }
  return session
}
```

#### 4.3.2 条件渲染

前端根据 `session.user.isStaff` 控制导航菜单和操作按钮的显示：

- 导航栏增加"管理"入口，仅 `isStaff=True` 时显示
- 任务看板增加"查看全部任务"开关，仅管理员可见

**再次强调**：前端权限仅为 UI 展示优化，所有实际数据访问权限由后端 API 校验。

### 4.4 用户管理

用户管理通过 Django Admin 完成，不在前端单独开发管理页面：

- 管理员登录 `http://localhost:8000/admin/` 进入后台
- 使用 `django-unfold` 主题提供的用户管理功能
- 可执行操作：查看用户列表、启用/禁用账号（`is_active`）、删除用户

## 九、已知问题和风险

### 9.1 需要修复的已有 Bug

在实现第一期功能前，需先修复以下已知问题：

1. **`backend/api/serializers.py` 第 99 行**：`validate_password()` 抛出的是 `django.core.exceptions.ValidationError`，但代码捕获的是 `rest_framework.exceptions.ValidationError`，导致 except 块永远不会触发
2. **`frontend/apps/web/lib/validation.ts`**：注册密码最小长度为 6，与后端 Django `MinimumLengthValidator` 默认值 8 不一致

### 9.2 风险点

| 风险 | 影响 | 应对 |
|------|------|------|
| 163 邮箱 SMTP 可能有发送频率限制 | 验证码发送失败 | 记录日志，前端展示友好错误提示 |
| next-intl 路由迁移需要移动所有页面文件 | 改动范围大 | 逐页迁移，每页迁移后验证 |
| User 模型 email 字段改为必填唯一 | 已有无 email 的用户数据需处理 | 迁移脚本为现有用户填充唯一占位 email |
| 验证码存在数据库中无加密 | 理论上数据库泄露可看到验证码 | 验证码 5 分钟短时效 + 单次使用，风险可接受 |
